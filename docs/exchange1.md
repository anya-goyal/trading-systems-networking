# Exchange 1 API Documentation
**Version:** 1.3  
**Protocols:** UDP (Market Data), TCP (Order Entry & Snapshots)

**IOG compatibility:** Order-entry and execution-report formats match the Internal Order Gateway wire format described in [`iocdocs.md`](iocdocs.md) §4.3–4.4.

---

## 1. General Conventions

* **Encoding**
  * *UDP (PITCH):* UTF-8 text, fields separated by pipe `|`.
  * *TCP (FIX):* ASCII tag=value pairs. **On the wire, fields are separated by SOH (Start of Header, ASCII `0x01`), not the pipe character.** Documentation often shows pipes for readability only.
* **TCP message framing:** Each logical FIX message includes tag **`10` (CheckSum)** as the **last field**. The stream is a sequence of such messages; there is **no** required trailing newline after a message. Parsers should locate `10=` and read the three-digit checksum, terminated by SOH, to delimit messages (same strategy as the IOG).
* **Pricing:**
  * *PITCH (UDP):* Prices are fixed-point integers (Price × 10000). E.g., `$150.25` is `1502500`.
  * *FIX (TCP):* Prices are decimal strings in tag `44`, e.g., `150.25` or `150` when whole.
* **Timestamps:** Unix epoch milliseconds in UDP PITCH payloads.

## 2. Market Data Feed (Simplified PITCH)
**Transport:** UDP Multicast/Broadcast  
**Direction:** Exchange → Subscribers  
**Description:** A low-latency, unacknowledged firehose of public order book events.

### Message Structure
All UDP messages follow a strict position-based format separated by `|`:
`Sequence_Number | Timestamp | Message_Type | [Payload]`

### Supported Message Types

#### System Event (`S`)
Indicates a change in the exchange's trading state.
* **Format:** `Seq | Time | S | EventCode`
* **Event Codes:** `O` (Open), `C` (Close), `H` (Halt)
* **Example:** `1|1711805400000|S|O`

#### Add Order (`A`)
Broadcasts a newly resting limit order on the visible book.
* **Format:** `Seq | Time | A | OrderID | Symbol | AssetType | Side | Shares | PriceInt`
* **Fields:**
  * `OrderID`: 64-bit integer generated by the exchange.
  * `AssetType`: String or character indicating the asset class (e.g., `EQ` for Equity, `OPT` for Options, `FUT` for Futures).
  * `Side`: `B` (Buy) or `S` (Sell).
* **Example:** `2|1711805400050|A|101|AAPL|EQ|B|100|1502500`

#### Order Executed (`E`)
Indicates a resting order was matched. (Note: Does not include Symbol; subscribers must maintain a local `OrderID` mapping).
* **Format:** `Seq | Time | E | OrderID | ExecutedShares`
* **Example:** `3|1711805400100|E|101|50`

#### Order Canceled (`X`)
Indicates a resting order was removed or reduced in size.
* **Format:** `Seq | Time | X | OrderID | CanceledShares`
* **Example:** `4|1711805400150|X|101|50`

## 3. Order Entry (FIX 4.2 over TCP)
**Transport:** TCP  
**Direction:** Bidirectional (IOG ↔ Exchange)  
**Description:** Reliable session for order entry and execution reports. The IOG is the **only** expected client on this port; it sends `49=IOG` and expects `56=IOG` on responses.

### 3.1 Wire format

* Between each `tag=value` pair, send **one SOH byte** (`\x01`).
* After all body fields (including the SOH following the last body tag), append **`10=NNN`** where `NNN` is three decimal digits: **(sum of all bytes in the message so far) mod 256**, zero-padded to width 3.
* Append a final **SOH** after the checksum value.
* **Display convention:** Examples below use `|` where the real byte is SOH, matching [`iocdocs.md`](iocdocs.md).

### 3.2 Tag dictionary (order entry and reports)

| Tag | Name | Type | Description |
| :--- | :--- | :--- | :--- |
| **`8`** | `BeginString` | String | Always `FIX.4.2`. |
| **`35`** | `MsgType` | String | `D` NewOrderSingle, `F` OrderCancelRequest, `8` ExecutionReport, `V` MarketDataRequest. |
| **`49`** | `SenderCompID`| String | Sender: `IOG` from client, `EXCH1` from exchange on reports. |
| **`56`** | `TargetCompID`| String | Target: `EXCH1` from client, `IOG` from exchange on reports. |
| **`10`** | `CheckSum` | String | **Required** on every FIX message. Three digits, see §3.1. |
| **`11`** | `ClOrdID` | String | Client order id from IOG. On **`35=F`**, this is the **ClOrdID of the live order to cancel** (IOG does not send tag `41`). |
| **`37`** | `OrderID` | String | Exchange-assigned order id (execution reports). |
| **`41`** | `OrigClOrdID`| String | *Not used* by the IOG; parsers should not require it for `35=F` from IOG. |
| **`55`** | `Symbol` | String | Ticker symbol (e.g., `AAPL`). |
| **`54`** | `Side` | Int | `1` = Buy, `2` = Sell. |
| **`40`** | `OrdType` | Int | `1` = Market, `2` = Limit. |
| **`38`** | `OrderQty` | Int | Total shares requested. |
| **`44`** | `Price` | String | Limit price as decimal text. The IOG sends this for both limit and market orders (e.g., `0` for market); the exchange may ignore for `40=1`. |
| **`59`** | `TimeInForce`| Int | **Optional** on inbound `35=D` from IOG. If absent, the exchange MUST behave as **`0` (Day / GTC-style day session)**. `3` = IOC when present. |
| **`150`**| `ExecType` | String | `0` New/Ack, `1` Partial fill, `2` Fill, `4` Canceled, `8` Rejected. |
| **`39`** | `OrdStatus` | String | Should be consistent with `150` (see FIX 4.2). |
| **`14`** | `CumQty` | Int | Total filled quantity. |
| **`31`** | `LastPx` | String | Last fill price (decimal text). |
| **`32`** | `LastQty` | Int | Last fill quantity. |
| **`58`** | `Text` | String | Optional reject / free text (parsed by IOG when present). |

### 3.3 Inbound to Exchange (from IOG)

#### New Order Single (`35=D`)

Typical field order from IOG: `8`, `35`, `49`, `56`, `11`, `55`, `54`, `40`, `38`, `44`, `10`.

* **Readable form (replace `|` with SOH on the wire):**
  `8=FIX.4.2|35=D|49=IOG|56=EXCH1|11=[ClOrdID]|55=[Symbol]|54=[Side]|40=[OrdType]|38=[Qty]|44=[Price]|10=[CHK]`
* **Example (pipe = SOH):**  
  `8=FIX.4.2|35=D|49=IOG|56=EXCH1|11=STRAT1-01|55=AAPL|54=1|40=2|38=100|44=150.25|10=...`  
  (`10` must be computed over the exact bytes through the SOH after `44=150.25`.)

#### Order Cancel Request (`35=F`)

IOG identifies the order to cancel **only** via tag **`11`** (same ClOrdId as the original `35=D`). Tags **`55`** and **`54`** are included. **Tag `41` is not sent.**

* **Readable form:**  
  `8=FIX.4.2|35=F|49=IOG|56=EXCH1|11=[ClOrdID]|55=[Symbol]|54=[Side]|10=[CHK]`

### 3.4 Outbound to IOG (ExecutionReport `35=8`)

Swap direction: `49=EXCH1`, `56=IOG`. The IOG requires tag **`11`**, and parses **`150`**, **`14`**, **`31`**, **`32`**, and optionally **`58`**. Every message MUST end with **`10` + SOH** so the IOG can frame the TCP stream.

* **Base shape:**  
  `8=FIX.4.2|35=8|49=EXCH1|56=IOG|11=[ClOrdID]|37=[ExchOrderID]|150=[ExecType]|39=[OrdStatus]|…|10=[CHK]`

* **Example (Ack):**  
  `8=FIX.4.2|35=8|49=EXCH1|56=IOG|11=STRAT1-01|37=999|150=0|39=0|10=...`

* **Example (full fill):**  
  `8=FIX.4.2|35=8|49=EXCH1|56=IOG|11=STRAT1-01|37=999|150=2|39=2|14=100|31=150.25|32=100|10=...`

* **Example (reject):** use **`150=8`** (Rejected), not `4`.  
  `8=FIX.4.2|35=8|49=EXCH1|56=IOG|11=STRAT1-02|37=1000|150=8|39=8|58=INVALID_ORDER|10=...`

* **Example (outright / cancel confirm):**  
  `8=FIX.4.2|35=8|49=EXCH1|56=IOG|11=STRAT1-01|37=999|150=4|39=4|10=...`

## 4. Snapshot Server (TCP Spin Port)
**Transport:** TCP  
**Direction:** Bidirectional (Data Handler ↔ Exchange)  
**Description:** Late-joining data handlers can request a resting-book snapshot. Use the **same FIX-style framing as §3** (SOH delimiters and tag `10` on FIX envelopes) so clients can share one parser.

#### Market Data Request (`35=V`)

Sent by the subscriber to request the current resting book.

* **Readable form:**  
  `8=FIX.4.2|35=V|49=[HANDLER_ID]|56=EXCH1|55=[Symbol]|10=[CHK]`  
  Use `ALL` in `55` to request all symbols.

#### Snapshot Response (Custom PITCH burst)

After accepting the request, the exchange may send **plain line-oriented text** (UTF-8, newline-terminated) for the snapshot body:

```text
SNAPSHOT_SEQ=[Current_UDP_Sequence]\n
[PITCH_A_Message_1]\n
[PITCH_A_Message_N]\n
END_SNAPSHOT\n
```

Each `[PITCH_A_Message_*]` line uses the **UDP PITCH** `A` format from §2 (pipe-separated). If a single TCP session interleaves FIX and this snapshot block, document session rules in the handler; a simple approach is a dedicated snapshot TCP port per connection policy.
